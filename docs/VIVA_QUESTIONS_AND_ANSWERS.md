# Viva Questions and Answers

Anticipated defense questions with honest, defensible answers. Cross-links
point to the full detail behind each short answer.

---

**Q: Why IRT instead of just percentage scoring?**
Percentage scoring treats every question as equally hard, which is false —
a 3-step logic puzzle and a "what's next in this sequence" question aren't
the same difficulty. IRT gives every question its own calibrated
difficulty and every student their own ability estimate, on the same
scale, so a score is a genuine measurement rather than a count.
[IRT_METHODOLOGY.md](IRT_METHODOLOGY.md)

**Q: Why the Rasch model specifically, not a 2- or 3-parameter model?**
It needs less response data per item to calibrate reliably — important
for a bank that keeps growing — and has the specific-objectivity property,
meaning ability estimates don't depend on which particular items were
used. Discrimination is still tracked, just as a separate diagnostic
rather than folded into the ability estimate.

**Q: How was the IRT engine validated — do you have evidence it actually
works?**
Monte Carlo simulation against synthetic data with known true parameters:
item-parameter recovery r = 0.991, person-parameter recovery r = 0.915.
Both are strong recovery correlations by published Rasch-simulation
benchmarks.

**Q: What does the ML model actually predict, and how confident should we
be in it?**
Exam readiness (a 4-class label + percentage), risk of dropping practice,
predicted next score, and predicted score change. Confidence should be
calibrated to the data: macro-F1 of 0.68 on held-out real+synthetic data,
and the model only ever informs (never gates) what a student can do next.
Every `/predict` response includes an explicit confidence-note disclaimer
distinguishing "model estimate" from "verified outcome."
[ML_READINESS_MODEL.md](ML_READINESS_MODEL.md)

**Q: Isn't training on OULAD (UK) and UCI (Portugal) data invalid for Sri
Lankan students?**
It's a real limitation, stated explicitly, not hidden. No dataset with
valid Sri Lankan exam-outcome ground truth exists. The response was
threefold: (1) use real data anyway for the features that generalize
across any learning context (score trends, consistency), (2) calibrate
the synthetic portion against real outcomes rather than guessing weights,
and (3) build the Sri Lankan-specific adaptation through local exam
structures, question content from real Sri Lankan reference materials, and
Sinhala support — channels other than the training data itself.
[DATASET_METHODOLOGY.md](DATASET_METHODOLOGY.md)

**Q: You added response-time tracking — did it improve the model?**
No, and that's reported honestly. A dedicated ablation study comparing 6
feature-set variants found no time-aware variant beat the current live
model; adding response-time features scored *below* the behaviour-only
variant. The live model was not swapped. This is a legitimate negative
result, not a failure to hide — the brief itself asked not to claim
improved accuracy until evaluation demonstrates it.
[TIME_AWARE_ANALYTICS.md](TIME_AWARE_ANALYTICS.md)

**Q: Why didn't you build "recommended study hours" or "best learning
strategy" as ML predictions, if the brief mentioned them?**
Because no dataset — real or synthetic — provides valid ground truth for
either. Building them as ML outputs would mean training a model against
either random synthetic labels or hoped-for values, which isn't
measurement, it's decoration. They're delivered instead as rule-based
`StudyPlanService` recommendations, explicitly not framed as ML outputs.

**Q: How do you know the AI-generated questions are actually correct?**
Two different mechanisms depending on the source. Most of the bank
(~6,500 questions) is generated by deterministic PHP code where the
correct answer is computed by a real solver function for that archetype —
never hand-asserted. Separately, LLM-drafted questions (Gemini or a
rule-based Mock fallback) always sit in a review queue; nothing reaches a
student until an admin explicitly approves it.
[QUESTION_GENERATION_SYSTEM.md](QUESTION_GENERATION_SYSTEM.md)

**Q: What happens to a student's readiness number when they don't have an
exam target?**
It's still computed — the same model, the same 43 features — but labelled
"Overall Cognitive Readiness" instead of implying a probability of passing
a specific exam. The distinction is applied at the presentation layer
based on whether an active, not-yet-due exam profile exists; it never
displays a fabricated placeholder number.

**Q: What happens once the exam date passes?**
The exam profile stops being shown as an active countdown. The student is
prompted once for a real outcome (attended / passed / score) — this
becomes genuine ground-truth data a future model retrain could use to
validate predictions against real outcomes, which nothing else in the
system currently provides. The prompt never blocks the student; skipping
it is a valid response.

**Q: Is the Sinhala translation reliable?**
Structurally validated (numeric-literal parity, option-count parity,
answer-key presence between EN/SI pairs) — explicitly *not* a claim of
deep semantic/NLP validation, stated as such in the validating service's
own documentation. Text corruption is prevented by a corpus-validation
process that checks every new string against forbidden codepoints and
requires new vocabulary to be explicitly reviewed.
[SINHALA_LANGUAGE_METHODOLOGY.md](SINHALA_LANGUAGE_METHODOLOGY.md)

**Q: How is student data privacy handled, especially with feedback?**
Feedback CSV exports never select user-identifying columns at all — not
redacted after the fact, simply never queried — so the anonymization is
structural, not a filter that could be forgotten. Synthetic demo accounts
and demo feedback are flagged (`is_demo_user`/`is_demo_feedback`) and
excluded from every research export by default, so screenshots/testing
data can never accidentally be presented as real research evidence.

**Q: What's the single biggest limitation you'd flag to an examiner
unprompted?**
The readiness model's ground truth is only ~46% real; the majority is
calibrated synthetic data. This is disclosed prominently rather than
buried, along with the population mismatch (UK/Portuguese training data
vs. Sri Lankan target users) — both are threats to external validity that
a future real-data retrain is the intended path to closing.

**Q: If you had another month, what would you build next?**
Complete formal model-registry versioning for the live model (currently a
real, documented gap — evaluation was run but never formally registered);
re-run the time-aware ablation once genuine (not synthesized)
response-time training data accumulates from real usage; and extend the
admin UI redesign to the remaining admin pages beyond the ones explicitly
scoped this session.
