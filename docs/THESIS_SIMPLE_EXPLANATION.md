# Thesis — Simple Explanation

A plain-language walkthrough of the whole project, written so you can
explain it confidently without reading code — start here, then dip into
the more technical docs (linked throughout) for anything you need to go
deeper on.

## What was built

HelaIQ is a web app for Sri Lankan government-exam candidates that
replaces static PDF question banks with an adaptive, measured, bilingual
practice platform. A student takes a placement test, gets daily practice
weighted toward their weak areas, sees a genuine readiness estimate
(not a guess), can take timed mock exams, plays cognitive-training
games, and reads self-learning notes — all in English or Sinhala.

## Why it was built this way

The original brief was flagged by the supervisor as "lacking novelty" for
being just a static question app. The response wasn't to add more
features randomly — it was to add **measurement and adaptation** at every
layer: measure ability properly (IRT, not raw percentages), measure
readiness with real evidence (a trained ML model, not a rule of thumb),
and adapt what's shown next to what's actually weak (weak-area weighting,
spaced repetition).

## How IRT works, in one paragraph

Instead of scoring a placement test as "12/20 correct," the platform gives
every question its own difficulty number and every student their own
ability number (theta), both on the same scale. The probability of
answering correctly depends only on the *gap* between the two — if
they're equal, it's a 50/50 question, exactly the intuitive definition of
"this is at your level." The test adapts live: after each answer, your
ability estimate updates, and the next question is chosen to be maximally
informative at your current estimate. Full detail:
[IRT_METHODOLOGY.md](IRT_METHODOLOGY.md).

## How the ML model works, in one paragraph

A gradient-boosted model (XGBoost, chosen after screening 9 algorithms)
takes 43 numbers describing a student — their ability, per-category
accuracy, practice consistency, self-reported motivation, and more — and
outputs a readiness percentage plus a plain-English explanation of why. It
was trained on a mix of real learning-analytics data (from a UK distance-
learning dataset, since no Sri Lankan exam-outcome dataset exists) and a
synthetic dataset whose weights were calibrated against that real data, not
guessed. Full detail: [ML_READINESS_MODEL.md](ML_READINESS_MODEL.md).

## What dataset was used, and why

The honest answer: no dataset perfectly matches this platform's actual
population (Sri Lankan government-exam candidates). Two real, public,
licensed datasets were used because they're the closest available signal
for how students' practice patterns predict outcomes — OULAD (Open
University, UK) and a UCI Portuguese-student dataset. Neither is Sri
Lankan; this mismatch is stated plainly as a limitation, not hidden. Full
detail: [DATASET_METHODOLOGY.md](DATASET_METHODOLOGY.md).

## How predictions work

When a student clicks "run prediction," Laravel builds their 43-value
feature vector from their *own real activity in the database* — it can't
be faked by the browser, because the browser never talks to the ML
service directly. The model returns a percentage, a label, and the top
reasons (via SHAP, a standard explainability technique that says which
input features pushed the score up or down). If the student has no active
exam target, this is framed as "Overall Cognitive Readiness"; if they do,
it's framed as "Readiness for {their exam}" — same model, different label,
because a student without an exam shouldn't be told a probability of
passing an exam that doesn't exist.

## How response time is used

Every question now records how long a student took to answer. This
enables a real, capped "speed-accuracy" score (a wrong answer always
scores zero regardless of speed; a slow-but-correct answer is mildly
penalized; there's no bonus for answering fast). A dedicated experiment
tested whether adding these response-time signals to the ML model actually
improves it — it didn't. That's reported honestly as a negative result,
not hidden, because a thesis claiming an improvement that doesn't hold up
under evaluation would be worse than reporting "we tried this, it didn't
help." Full detail: [TIME_AWARE_ANALYTICS.md](TIME_AWARE_ANALYTICS.md).

## How Sinhala is handled

Every student-facing string exists in English and Sinhala. Because Sinhala
script is easy to corrupt when typed from memory (a mistake this project
actually made twice, both times caught before it shipped), a validation
tool checks every new Sinhala string for corrupted characters and requires
new vocabulary to be explicitly logged as reviewed. Full detail:
[SINHALA_LANGUAGE_METHODOLOGY.md](SINHALA_LANGUAGE_METHODOLOGY.md).

## How AI question generation works

Most questions in the bank (~6,500) are generated by deterministic PHP
code, not an LLM — each archetype (blood relations, seating arrangements,
Venn puzzles) has a real solver function that computes the correct answer,
so answers are never asserted, only computed. A separate pipeline lets an
admin generate draft questions via Gemini (or a rule-based Mock fallback
when no API key is configured), but nothing an AI drafts reaches students
without an admin reviewing and approving it first. Full detail:
[QUESTION_GENERATION_SYSTEM.md](QUESTION_GENERATION_SYSTEM.md).

## Limitations, stated plainly

- The readiness label is only ~46% grounded in real outcome data; the rest
  is calibrated synthetic data.
- No dataset represents the platform's actual target population.
- Time-aware features don't currently improve the model (negative
  result, see above).
- `model_registry.py`'s formal versioning was never actually run for the
  currently-live model — it's a real gap, documented, not swept under the
  rug.
- A handful of pre-existing data-quality issues (some duplicate/redundant
  question rows, 14 study notes with mismatched taxonomy tags) are known
  and flagged, not silently fixed with a risky migration.

## Future improvements

- Retrain on real HelaIQ usage data once enough students have used the
  platform, to replace or supplement the synthetic training portion.
- Re-run the time-aware ablation once real (not synthesized) response-time
  training data exists.
- Complete `model_registry.py` version registration for the live model.
- Extend the admin UI redesign to the remaining admin pages (a documented,
  deliberate scope cut, not an oversight).
