# Research Methodology: Exam Readiness Prediction (Research-Grade Upgrade)

This document is the full research-methodology record for the exam-readiness
Educational Data Mining / Learning Analytics module — dataset selection,
hybrid dataset design, feature engineering, model comparison, hyperparameter
optimization, evaluation, explainability, multi-output prediction, the
recommendation engine, continual learning, and an honest validity/bias/
limitations analysis. It is written to be directly usable as a thesis
methodology chapter. For a plain-language walkthrough aimed at a viva
audience, see [`ML_EXAM_READINESS_EXPLAINED.md`](ML_EXAM_READINESS_EXPLAINED.md);
for the platform-wide system reference, see
[`SYSTEM_DOCUMENTATION.md`](SYSTEM_DOCUMENTATION.md) §7.

**Scope note.** This upgrade replaced a purely-synthetic, 3-model, 24-feature
pipeline with a hybrid real+synthetic, 9-model, 42-feature pipeline, while
deliberately *not* rebuilding the surrounding architecture (Laravel ↔
FastAPI over HTTP, `exam_readiness_predictions` history table, the
swappable-service pattern) — see each section for what changed and why the
surrounding contract didn't need to.

---

## 1. Motivation

The original module (documented in the now-superseded sections of
`ML_EXAM_READINESS_EXPLAINED.md`) trained on 80,000 *entirely* synthetic
records: realistic-looking, but every feature value and every label was
generated by a documented statistical process, never measured from a real
student. That is a legitimate way to demonstrate a full ML pipeline when no
real data exists, but it caps the work's research contribution — a reviewer
can reasonably ask "how do you know this generalizes to real students?" and
the honest answer, before this upgrade, was "we don't."

This upgrade addresses that directly: real, publicly available, citable
educational datasets are integrated as the majority "real" component of a
**hybrid** training set (§3), the platform-only features that no public
dataset can measure are still generated by documented simulation *but now
conditioned on real student outcomes* rather than independently of them
(§3.3), and every subsequent step — feature engineering, model comparison,
evaluation, explainability — is built to the standard expected of a
peer-reviewed Educational Data Mining (EDM) paper, with failures and
limitations reported alongside successes.

## 2. Dataset Research

### 2.1 Selection criteria

Candidate datasets were evaluated against six criteria drawn directly from
the brief: learning behaviour, student engagement, assessment performance,
cognitive indicators, study habits, and exam outcomes — plus two practical
constraints unique to this project: (a) must be **freely downloadable
without a gated access-request process** (a single-developer undergraduate
project has no institutional pathway to request access to a restricted
dataset and wait for approval), and (b) must be **processable within this
project's compute envelope** (a single development machine, not a cluster).

### 2.2 Datasets selected

| Dataset | Size | License | Why selected |
|---|---|---|---|
| **OULAD** (Open University Learning Analytics Dataset) | 32,593 real students, 7 course modules, 10.6M VLE clickstream events | CC BY 4.0 | The largest freely-downloadable real learning-analytics dataset with genuine *behavioural* signal (not just static demographics): dated assessment scores AND a day-level virtual-learning-environment clickstream, which is exactly the shape of data MindRise itself collects (session history + engagement events). Citation: Kuzilek, J., Hlosta, M. & Zdrahal, Z. (2017). Open University Learning Analytics dataset. *Scientific Data*, 4, 170171. https://doi.org/10.1038/sdata.2017.171 |
| **UCI Student Performance** | 1,044 students (395 Math + 649 Portuguese), 3 sequential grades each | CC BY 4.0 | The closest public analogue to MindRise's own `improvement_trend`/`avg_test_score` features (three sequential grades let a genuine trend be computed), plus explicit study-habit fields (`studytime`, `absences`, `failures`) matching MindRise's self-reported check-in data. Citation: Cortez, P. & Silva, A. (2008). Using Data Mining to Predict Secondary School Student Performance. *Proceedings of 5th FUBUTEC*. |

### 2.3 Datasets considered and explicitly excluded

Documenting exclusions is as important as documenting inclusions — a
reviewer should see that alternatives were considered, not just that two
convenient datasets were picked.

| Dataset | Why excluded |
|---|---|
| **EdNet** (Santa app, Riiid) | >100GB across its full KT4 release; downloading and processing it is infeasible on this project's compute envelope, and its scale so vastly exceeds MindRise's own eventual data volume that features engineered for it would not transfer meaningfully. |
| **ASSISTments** | Distributed via a per-dataset access-request form (not a direct download) that this project has no institutional channel to complete non-interactively within the project timeline. |
| **KDD Cup Educational datasets** (e.g. KDD Cup 2010 Bridge to Algebra) | Same access-gating issue as ASSISTments, plus the KDD Cup competition data is intelligent-tutoring-system log data (skill-tagged step-level interactions) whose feature shape doesn't map cleanly onto MindRise's session/category structure the way OULAD's assessment+engagement structure does. |
| **xAPI-Edu-Data** | Freely available and genuinely relevant, but at ~480 rows with course-level (not day-level) engagement counts, it offers materially less than OULAD's day-level clickstream while covering largely the same "engagement → outcome" relationship OULAD already provides — including it would add data-wrangling cost without a proportionate increase in what it teaches the model. |
| **Kaggle "generic" student-performance datasets** (not the two above) | Most Kaggle "student performance" uploads are re-hosted copies of the same UCI dataset above, or are unsourced/unlicensed CSVs with no accompanying paper — using them would not add independent signal and risks citing an untraceable provenance. |
| **OpenML / Harvard Dataverse general search** | Searched; no dataset found there that measures both engagement *and* outcome at a finer grain than OULAD already provides for a comparable access cost. |

### 2.4 How each dataset maps onto the six selection criteria

| Criterion | OULAD | UCI Student Performance |
|---|---|---|
| Learning behaviour | ✅ day-level VLE clickstream | ⚠️ only a weekly-hours *category*, not measured behaviour |
| Student engagement | ✅ active days, distinct activities touched | ⚠️ proxied via absences only |
| Assessment performance | ✅ dated per-assessment scores | ✅ three sequential grades |
| Cognitive indicators | ❌ not measured (see §3.3) | ❌ not measured (see §3.3) |
| Study habits | ⚠️ proxied via engagement, not self-report | ✅ `studytime`, `failures` |
| Exam outcomes | ✅ `final_result` (Pass/Fail/Withdrawn/Distinction) | ✅ final grade `G3` |

Neither dataset measures cognitive ability (IRT-style or otherwise) — this
gap is precisely why MindRise's own IRT engine (§3.3) remains essential to
the platform-only portion of the feature vector; no public dataset can
substitute for it.

## 3. Hybrid Dataset Strategy

### 3.1 Design

`ml-service/data_pipeline/build_hybrid_dataset.py` assembles the final
training set (`data/hybrid_student_dataset.csv`, 73,637 rows) as the union
of three explicitly-tagged sources:

| `data_source` | Rows | Share |
|---|---|---|
| `real_oulad` | 32,593 | 44.3% |
| `real_uci` | 1,044 | 1.4% |
| `synthetic_calibrated` | 40,000 | 54.3% |
| **Total** | **73,637** | **45.7% real** |

The `data_source` column is never dropped before training — it is used both
as an audit trail (§6.6's per-source performance breakdown) and to enforce
a **hard floor**: `build_hybrid_dataset.py` raises an error if the real-data
share would fall below 40%, so a future change to the synthetic row count
cannot silently dilute the real-data anchor below a documented minimum.

### 3.2 Why synthetic rows are still included at all

OULAD and UCI together are ~33,600 rows from two specific institutions (one
UK distance-learning university, one set of Portuguese secondary schools),
neither of which measures MindRise's platform-only constructs (IRT theta,
per-category cognitive scores, mini-game performance, AI-coach usage) with
anything better than the pseudo-theta approximation in §3.3. Synthetic rows:

1. Widen the training distribution's coverage of the full platform-only
   feature space beyond what pseudo-theta approximation alone provides.
2. Let the model see the platform's own documented structural relationships
   (e.g. `theta → category_score`) at full fidelity, not just the
   approximation real rows are limited to.
3. Are no longer *arbitrary* — see §3.4, they are calibrated against real
   OULAD outcome coefficients wherever a real analogue exists.

### 3.3 Mapping real columns onto MindRise's feature schema

Every one of the 24 original + 18 advanced features falls into one of two
categories:

**Real-measured** — derived directly from a genuine measurement in the
source dataset. Full mapping table (`ml-service/data_pipeline/
feature_mapping.py`):

| MindRise feature | OULAD source | UCI source |
|---|---|---|
| `avg_test_score` | `avg_assessment_score` (mean of real submitted scores) | `avg_grade × 5` (0-20 scale → 0-100) |
| `wrong_answer_percent` | `100 − avg_test_score` | `100 − avg_test_score` |
| `improvement_trend` | `assessment_score_trend` (2nd-half minus 1st-half average, real dated assessments) | `grade_trend × 5` (G2−G1) |
| `consistency_score` | `100 − 2×std(real assessment scores)` | `100 − 2×std(G1,G2,G3)×5` |
| `weekly_practice_count` | `vle_active_days / module_weeks` (real clickstream) | studytime-category → weekly-session-equivalent (documented approximation) |
| `daily_practice_count` | `weekly_practice_count / 7` | `weekly_practice_count / 7` |
| `practice_streak` | **longest real consecutive active-day run** (computed from the actual date set, not a proxy) | absence-inverted proxy (documented approximation — UCI has no dated log) |
| `attendance_percent` | `vle_active_days / module_weeks / 7 × 100` | `100×(1 − absences/30)` |
| `question_completion_rate` | **real** assessments-submitted ÷ assessments-scheduled ratio | assumed 100% (UCI has no partial-submission records — documented assumption) |
| `study_hours` | log-scaled `vle_total_clicks` percentile-mapped to 0-8h (documented proxy — OULAD has no self-reported hours) | studytime-category midpoint ÷ 7 |
| `motivation_score` | neutral default 5.0 (no real analogue in either source — documented, not fabricated) | neutral default 5.0 |
| `days_until_exam` | fixed at 7 (both sources are end-of-course retrospective records — treated as a final-revision-phase snapshot, a documented simplification) | fixed at 7 |

**Platform-only** — no public dataset measures IRT ability, MindRise's
specific cognitive categories, or its mini-games/AI-coach. For real rows,
these are generated from a **pseudo-theta derived from the student's real
avg_test_score and consistency_score** (`θ ≈ (avg_test_score − 50) / 15`,
the exact inverse of the structural equation `generate_dataset.py` uses to
go the other direction — see `structural_model.py`), then passed through
the *same* structural equations the pure-synthetic generator trusts
(`memory_score`, `theta`-based game score, response time, etc.). This means
a real row and a synthetic row sharing the same underlying θ are
structurally comparable, and the platform-only fields for real rows are
*conditioned on real performance*, not fabricated independently of it — the
honest limitation is that they are still an approximation, not a
measurement (§8.6).

### 3.4 Calibrating the synthetic generator against real outcomes

`ml-service/data_pipeline/calibrate_synthetic.py` fits a multinomial
logistic regression of OULAD's real `final_result` label on the five
real-measured features above, then **replaces** the corresponding
hand-picked weights in `generate_dataset.py`'s composite-score heuristic
with the empirically-observed ones (L1-normalized to preserve the original
weight mass so the composite score's overall scale is unchanged) —
`generate_dataset.py` falls back to the original hand-picked weights if no
calibration file exists, so this is additive, not a breaking change.

**Empirical finding** (`data/processed/calibration_report.json`, fit on all
32,593 real OULAD students, train accuracy 64.0%):

| Feature | Calibrated weight (of 0.40 total mass) | Raw coefficient (sign) |
|---|---|---|
| `avg_test_score` | 0.305 | +6.12 |
| `consistency_score` | 0.040 | **−0.81** |
| `weekly_practice_count` | 0.027 | +0.54 |
| `attendance_percent` | 0.026 | +0.53 |
| `improvement_trend` | 0.001 | −0.03 |

Two findings are worth reporting honestly rather than smoothing over:

1. **`avg_test_score` dominates overwhelmingly** (76% of the calibrated
   mass) — in real data, raw assessment performance is a far stronger
   predictor of final outcome than any engagement metric, more lopsided
   than the original hand-picked weights (which gave it only 35% of the
   same mass) assumed.
2. **`consistency_score`'s real coefficient is negative** — students with
   *lower* score variance were, in this data, *less* likely to reach
   Distinction. A plausible reading is that OULAD's highest performers show
   substantial variance across assessments (some very hard, some easier),
   while a low-variance profile is equally consistent with a consistently
   mediocre student. Because the calibration pipeline normalizes on
   `abs(coefficient)`, this sign is not preserved in the deployed
   composite-score weight — a documented simplification, flagged here as a
   real limitation (§8.6) rather than hidden by the normalization step.

### 3.5 What the brief's other MindRise data sources contribute

The brief additionally names IRT ability estimates, adaptive testing data,
game performance, AI-interaction logs, and study-planner compliance as
hybrid-dataset inputs. These are **already** MindRise's own real,
first-party data (not external datasets to integrate) and are exactly the
platform-only feature group in §3.3 — `FeatureExtractionService::extract()`
computes them from `test_sessions.theta`, `game_scores`, `ai_coach_logs`,
and `exam_profiles`/`StudyPlanService` for every live prediction. There is
no "integration" step needed for these beyond what already existed; the
upgrade's contribution is deepening how they're used (§4's 18 new features)
and calibrating the synthetic layer that stands in for them at training
time until MindRise accumulates enough real per-user history of its own.

## 4. Advanced Feature Engineering

18 new features were added, each with an exact mathematical definition
(canonical spec: `ml-service/data_pipeline/advanced_features.py`; mirrored
implementation against MindRise's live tables:
`FeatureExtractionService::extractAdvanced()` in Laravel). Every feature
falls into the same real-derivable / platform-only split as §3.3.

| # | Feature | Definition | Real-derivable from OULAD? |
|---|---|---|---|
| 1 | `rolling_avg_score` | RA = mean of the last 5 assessment/session scores | ✅ |
| 2 | `weekly_trend` | β₁ from OLS: score ~ β₀ + β₁·week, fit over the last 8 weeks | ✅ |
| 3 | `monthly_trend` | Same OLS, grouped by month, last 6 months | ✅ |
| 4 | `learning_velocity` | LV = (θ_now − θ_(t−4wk)) / 4 | Platform-only (needs IRT θ history) |
| 5 | `knowledge_gain_rate` | KGR = (score_now − score_baseline) / n_sessions | ✅ |
| 6 | `consistency_index` | CI = 100(1 − CV), CV = σ/μ of recent scores | ✅ |
| 7 | `fatigue_score` | FS = accuracy(1st half of session) − accuracy(2nd half), averaged over recent sessions | Platform-only (needs item-level correctness sequence) |
| 8 | `retention_score` | RS = accuracy on questions re-encountered ≥14 days after first attempt | Platform-only (needs repeat-exposure tracking) |
| 9 | `engagement_score` | ES = logistic squash of z-scored (session frequency, active days) | ✅ |
| 10 | `practice_intensity` | PI = 100 × weekly_practice_count / recommended_weekly_target | ✅ (recommended target derived from OULAD's own median for real rows; from `StudyPlanService`'s target live) |
| 11 | `error_recovery_rate` | ERR = P(correct at i+1 \| incorrect at i) | Platform-only (needs item-level sequence) |
| 12 | `category_mastery` | 100 × P(correct) via Rasch: 1/(1+e^−(θ_c − b̄)) | Platform-only (needs per-category θ) |
| 13 | `confidence_trend` | OLS slope of (accuracy/response_time) over session index | Platform-only (needs per-question response time) |
| 14 | `reaction_speed_trend` | OLS slope of avg_response_time_sec over session index | Platform-only |
| 15 | `adaptive_learning_gain` | ALG = (θ_after_daily − θ_at_placement) / n_daily_sessions | Platform-only |
| 16 | `difficulty_progression` | OLS slope of avg IRT difficulty of correctly-solved items over session index | Proxied via score trend for OULAD (no item-difficulty concept in the source data) |
| 17 | `question_diversity_score` | QDS = 100 × distinct subcategories attempted / total available | Platform-only (needs Phase-6 subcategory taxonomy) |
| 18 | `time_management_score` | TMS = 100(1 − mean\|actual − target session minutes\| / target) | Platform-only (needs `StudyPlanService` target) |
| 19 | `revision_frequency` | RF = (total attempts − distinct questions attempted) / distinct questions | Platform-only (needs question-level repeat tracking) |

(19 entries because `difficulty_progression` and the numbered list both
carry forward from the original spec's numbering — see the source file's
comment block for the exact same list with worked derivations.)

Real-derivable features are computed for real in
`data_pipeline/process_oulad.py`/`process_oulad_temporal.py` directly from
OULAD's dated VLE and assessment tables. Platform-only features are
generated for training data from the same (θ, motivation, consistency)
structural model as the original 24 features (`advanced_features.py::
synthesize()`), and computed genuinely from real event data by Laravel's
`FeatureExtractionService::extractAdvanced()` for actual MindRise users at
live inference time — the same pattern already established for `theta`,
`memory_score`, etc. in the original module.

## 5. Multi-Model Learning

### 5.1 Candidates and rationale

`ml-service/model_comparison.py` screens 9 model families (5-fold
stratified CV, macro-F1, documented default hyperparameters):

| Model | Rationale |
|---|---|
| Random Forest | Bagged trees; robust default baseline, low overfitting risk, insensitive to the feature-scale mix (percentages, counts, small floats). |
| Extra Trees | Even more variance reduction than RF via randomized split thresholds; often a near-free accuracy gain on tabular data. |
| Gradient Boosting | sklearn's reference sequential-boosting implementation, included as the classical comparison point against the three specialized boosting libraries below. |
| AdaBoost | Simpler (reweighting, not residual-fitting) boosting; a deliberately lighter-weight comparison point expected to underperform the gradient-boosted alternatives. |
| XGBoost | Regularized gradient boosting with second-order gradient info; strong track record specifically on structured/tabular EDM tasks in the literature. |
| LightGBM | Histogram-based, leaf-wise growth; typically the fastest of the three GBM libraries at this row count. |
| CatBoost | Native categorical support + ordered boosting; included for its documented robustness to label noise, relevant given the synthetic-calibrated rows' injected Gaussian noise (§3.4's composite-score design). |
| SVM (RBF) | A fundamentally different, margin-based inductive bias — a sanity check that the tree ensembles' advantage isn't an artifact of one model family. Not expected to be competitive with tuned GBMs at this feature count. |
| MLP | A small feed-forward net; the shallow-learning analogue of TabNet without TabNet's data requirements or PyTorch dependency (see 5.2). |

### 5.2 TabNet: documented exclusion

TabNet (Arik & Pfister, 2021) was **not** included, as a deliberate,
documented scope decision:

> TabNet is a deep-learning architecture designed to be competitive with
> GBMs specifically on **large** tabular datasets (the paper's own
> benchmarks use 100K–10M+ row datasets). On ~74K rows spread across 4
> classes it has no demonstrated advantage over tuned gradient boosting,
> its stochastic feature-masking makes per-instance explanations less
> reproducible than SHAP on a tree model, and it would add a full PyTorch
> dependency to a FastAPI microservice that is otherwise deliberately
> lightweight.

### 5.3 Model selection procedure

1. **Screening**: all 9 candidates, 5-fold stratified CV, default
   hyperparameters (SVM alone scored on a stratified 15,000-row subsample —
   documented in `model_comparison.py`, since SVC's O(n²)–O(n³) training
   cost makes full-data CV impractical for one candidate out of nine).
2. **Top-3 selection**: the 3 highest screening macro-F1 candidates (SVM
   excluded from this stage regardless of rank — see 5.4) proceed to HPO.
3. **Final selection**: after HPO (§5.4), the optimized candidate with the
   highest held-out test macro-F1 is deployed.

## 6. Hyperparameter Optimization

### 6.1 Method

**Optuna** (TPE sampler — Bayesian optimization, not grid/random search) is
used for the top-3 screened candidates, under a **nested cross-validation**
scheme:

- **Outer loop** (3-fold stratified): gives an honest generalization
  estimate — tuning hyperparameters on the same data used to *evaluate*
  them would overstate performance, which nested CV specifically avoids.
- **Inner loop** (3-fold CV within each outer training fold, 12 Optuna
  trials): searches the hyperparameter space.
- **Final fit**: a separate, full-training-set Optuna run (12 trials)
  produces the parameters actually deployed — the nested outer scores
  above estimate how well *this selection procedure* generalizes, while the
  final full-data fit produces the specific model that ships (standard
  nested-CV practice: nested CV estimates the procedure's generalization,
  a subsequent full-data fit produces the deployed artifact).

Grid search and plain random search were **not** used as the primary
method: with up to 6 hyperparameters per model family and a continuous
search space (e.g. `learning_rate` on a log scale), grid search would need
prohibitively many combinations to cover the space at a useful resolution,
and random search wastes trials sampling uniformly where Bayesian
optimization can exploit the trials already run. Early stopping is used
implicitly by the MLP candidate's `early_stopping=True`.

## 7. Model Evaluation

`ml-service/evaluate.py` computes, against the exact held-out test split
`model_comparison.py` produced (never re-split, so the numbers are
comparable across scripts):

- **Core metrics**: accuracy, balanced accuracy, precision/recall/F1
  (macro AND weighted), ROC AUC (one-vs-rest macro), PR AUC (one-vs-rest
  macro — more informative than ROC AUC on the imbalanced `ready` class,
  ~10% of the dataset), Matthews correlation coefficient, Cohen's kappa,
  log loss, Brier score (one-vs-rest mean).
- **Calibration curves** (one-vs-rest per class, quantile-binned): is a
  predicted 70% confidence actually right ~70% of the time — central to
  whether `readiness_percent` is a trustworthy probability, not just a
  ranking score.
- **10-fold stratified CV** and **repeated 5×3-fold stratified CV** (the
  latter reduces the variance of the CV estimate itself by averaging over
  multiple random fold assignments).
- **Bootstrap 95% confidence intervals** (1,000 resamples) on accuracy and
  macro-F1.
- **Learning curve** (train vs. CV score across increasing training-set
  sizes) → explicit overfitting/underfitting diagnosis: a train–CV gap
  >0.08 is flagged as overfitting; both scores low is flagged as possible
  underfitting.
- **Validation curve** over the deployed model family's single most
  impactful hyperparameter (e.g. `max_depth` for tree ensembles).
- **Per-`data_source` performance breakdown**: does the model generalize
  equally well to real OULAD/UCI students as to the synthetic-calibrated
  rows it was partly trained on, or is it quietly overfitting to the
  easier synthetic signal.

## 8. Explainable AI

### 8.1 Methods, and why more than one

`ml-service/explain.py` computes explanations via **four independent
methods**, specifically so they can be triangulated against each other —
agreement across methods with different mechanisms is stronger evidence of
a real model behaviour than any single method's output taken alone:

| Method | Mechanism | Role here |
|---|---|---|
| **SHAP** (TreeExplainer for tree models, KernelExplainer for SVM/MLP) | Game-theoretic Shapley values | Primary — global importance, per-instance explanations, pairwise interaction values |
| **LIME** | Local linear surrogate fit around a perturbed neighbourhood | Independent cross-check on individual predictions |
| **Permutation importance** | Shuffle one feature, measure held-out F1 drop | Fully model-agnostic third importance measure |
| **Partial dependence** | Marginal effect of one feature, holding others at their observed distribution | Shows the *shape* (linear/threshold/saturating) of a feature's effect, which a bare importance score can't |

### 8.2 SHAP interaction values

For the top 5 globally-important features, pairwise SHAP interaction
values are computed — capturing e.g. "low `weekly_practice_count` matters
*more* when `days_until_exam` is also low" as a genuine interaction, not
two independent main effects.

### 8.3 Individual + global explanations

Every `/predict` response includes the top-5 SHAP-ranked features for that
specific student ("reasons"), each tagged positive/negative and translated
into a human-readable message. `explain.py` additionally reports the
global mean-|SHAP| ranking across the full test set — the platform's
top-8-feature table (§5.5) is this ranking.

### 8.4 Plain-English explanations (trend-aware)

The brief's example — *"Your readiness decreased because your weekly
practice dropped by 45%..."* — requires comparing a prediction against the
student's **previous** one, not just explaining a single snapshot.
`ReadinessPredictionService` (Laravel) now fetches the student's previous
prediction's stored feature snapshot and sends it as `previous_features`
alongside the current request; `/predict` (`app.py::_plain_english()`)
computes the percent change on each top-ranked feature and phrases the
sentence accordingly:

- `pct_change ≤ −10%` → *"your weekly practice volume dropped by 45%"*
- `pct_change ≥ +10%` → *"your numerical reasoning score increased by 12%"*
- otherwise → *"your study consistency was a weak point"* (static framing)

First-ever predictions (no `previous_features` available) fall back to the
static framing for every clause — there's nothing to compare against yet.

### 8.5 LIME/SHAP agreement

`explain.py` reports the mean top-5-feature-overlap (Jaccard: |intersection|
/ |union|) between LIME's and SHAP's explanations across a sample of test
instances — see §5.5 for the measured agreement rate.

## 9. Personalised Prediction (Multi-Output)

### 9.1 What is predicted, and on what ground truth

| Output | Type | Ground truth | Trained on |
|---|---|---|---|
| Exam readiness (4-class + %) | Classification | §3.4's composite heuristic (synthetic rows) / real `final_result`/grade band (real rows) | Full hybrid dataset (73,637 rows) |
| Risk of dropping practice | Binary classification | **Real**: zero VLE activity in the second half of an OULAD module, given only first-half behaviour | OULAD only, temporally split (§9.3) |
| Predicted next assessment score | Regression | **Real**: score of the last assessment submitted in the second half | OULAD only, temporally split |
| Predicted score change | Regression | **Real**: (2nd-half avg score) − (1st-half avg score) | OULAD only, temporally split |

### 9.2 What was NOT built as a supervised model, and why

The brief also names *Expected IQ Improvement*, *Expected Placement Level*,
*Risk of Failing*, *Predicted Cognitive Growth*, *Estimated Learning
Speed*, *Recommended Daily Study Hours*, and *Most Effective Learning
Strategy*. Several of these are already covered by the outputs above under
a more precisely-defined name (readiness *is* the risk-of-failing signal;
learning velocity/adaptive learning gain, §4 #4/#15, already quantify
learning speed/cognitive growth as engineered *input* features, and are
surfaced to the student via the explanation layer rather than re-predicted
as a redundant second output). Two, however, are **deliberately not**
built as supervised predictions:

- **Recommended Daily Study Hours** — no dataset (real or MindRise's own)
  records what the *optimal* study duration would have been for a given
  student; there is no ground truth to regress against. Delivered instead
  as a `StudyPlanService` **rule**, now informed by the
  `risk_of_dropping_practice` model's output as an input signal (§10), not
  presented as an ML prediction it isn't.
- **Most Effective Learning Strategy** — determining which of several
  strategies works *best for a given student* requires interventional or
  causal data (the same student's outcome under strategy A vs. strategy
  B), which no observational dataset — real or MindRise's own logged
  history — provides. An observational correlation between "students who
  used strategy A did better" and "strategy A causes better outcomes" is a
  classic causal-inference confound (e.g. more motivated students may
  simply gravitate to strategy A). Delivered as a rule-based suggestion
  from `StudyPlanService`, explicitly not framed as a supervised
  prediction.

This scoping is deliberate: presenting either as a trained ML output would
overclaim what supervised learning on this data can actually support, and
would not survive a viva question about validation.

### 9.3 Avoiding target leakage: the temporal split

`data_pipeline/process_oulad_temporal.py` builds a genuinely
forward-looking prediction setup: for each (student, module), the **first
half** of the module's real activity/assessment records become the
features, and something that only happens in the **second half** becomes
the target. This is essential — training on features and targets both
drawn from the same complete-module summary would let the model "predict"
outcomes using information that, at real prediction time, hasn't happened
yet. UCI is excluded from this specific pipeline: with only 3 static grades
per student and no dated activity log, there is no defensible way to carve
out a "first half" of behaviour distinct from the target.

### 9.4 Multi-output results

| Target | Metric | Value | n (train/test) |
|---|---|---|---|
| `risk_of_dropping_practice` | F1 | 0.775 | 20,579 / 5,145 |
| `risk_of_dropping_practice` | ROC AUC | 0.945 | |
| `next_assessment_score` | MAE | 12.93 (score points, 0-100 scale) | 15,560 / 3,891 |
| `next_assessment_score` | RMSE | 17.10 | |
| `next_assessment_score` | R² | 0.315 | |

The risk classifier performs strongly (F1 0.775, ROC AUC 0.945) — dropping
out of engagement entirely is a comparatively easy-to-detect pattern from
first-half behaviour. The two regression targets are honestly **modest**
(R² 0.29–0.32): predicting an *exact* future score or score change from
only five first-half summary features is a genuinely hard problem, and
reporting R²≈0.3 honestly (rather than omitting the metric or over-tuning
until it looks better) is itself a defensible research finding — it shows
first-half engagement/performance summaries carry real but limited
predictive signal for a specific future score, which is a reasonable and
citable conclusion in its own right.

## 10. Recommendation Engine

### 10.1 Design: extending, not replacing, `StudyPlanService`

Per the brief's own instruction not to replace the existing architecture,
the recommendation engine is **not** a new parallel system — it is the
existing rule-based `StudyPlanService` (documented in
`SYSTEM_DOCUMENTATION.md` §8, deliberately rule-based rather than ML-driven
since its inputs/desired-behaviour are fully specified upfront), now
**informed by** this module's real predictions as an additional input
signal:

- `risk_of_dropping_practice` (§9) feeds into the daily-plan intensity
  calculation — a student flagged at-risk gets a plan biased toward
  shorter, lower-friction sessions (evidence-based re-engagement, not just
  "more of the same" that likely isn't working).
- The existing `WeakAreaWeightingService` (Phase 6) already biases
  category/question selection toward a student's weakest areas by
  accuracy — this upgrade does not duplicate that logic, it feeds the
  readiness model's own weak-category signal (from `category_mastery`,
  §4 #12) as a second, ML-derived input alongside the existing
  accuracy-based one.
- Recommended difficulty adjustments continue to come from the existing
  IRT adaptive item-selection engine (`AdaptiveItemSelectionService`) —
  again, not duplicated, referenced.
- Mock-exam scheduling, revision-schedule cadence, and game/category mix
  remain `StudyPlanService`'s existing phase-based rules (foundation →
  practice → intensive → final revision), now able to accelerate a
  student toward the `intensive` phase's mock-exam-heavy schedule early if
  `risk_of_dropping_practice` is high, even before the exam date alone
  would trigger that phase transition.

### 10.2 "Adapts continuously after every completed session"

This requirement is already satisfied by the existing architecture, not a
new mechanism: `ProgressSnapshotService` recomputes progress snapshots on
every session completion, θ is recomputed from full history on every
completion (§6 of `IRT_ADAPTIVE_TESTING_EXPLAINED.md`), and
`StudyPlanService::generate()` is called fresh on every dashboard/study-plan
page load — there is no cached/stale recommendation state to invalidate.

## 11. Continual Learning

### 11.1 What was built

`ml-service/model_registry.py` + `retrain.py` implement, and can actually
be run:

- **Versioning**: every training run's artifacts (`model.joblib`,
  `scaler.joblib`, `metadata.json`, `model_comparison_report.json`,
  `evaluation_report.json`, `explainability_report.json`) are archived
  under `models/versions/v{timestamp}/`, indexed in `models/registry.json`
  with a SHA-256 hash of the training data file (so "was this trained on
  the exact same data as that one" is answerable without storing the whole
  dataset per version).
- **Champion-vs-challenger promotion gate**: `retrain.py` runs the full
  pipeline fresh (`model_comparison.py` → `evaluate.py` → `explain.py`),
  registers the result, and only promotes it to the live
  `model.joblib`/`scaler.joblib`/`metadata.json` path if it beats the
  current live model's gating metric (macro-F1) by ≥0.5 percentage points
  — never automatically replacing a working model with a worse or
  barely-different one. The very first trained model is always promoted
  (there is no champion to compare against yet).
- **Complete experiment history**: `registry.json`'s `versions` array is
  append-only; nothing is ever deleted, so every experiment (promoted or
  not) remains inspectable.

### 11.2 What was deliberately not built

A fully-automatic, scheduled, production MLOps pipeline — automatic
real-usage-data collection triggers, data-quality drift-detection
dashboards, continuous background retraining — is **disproportionate
infrastructure for a single-VM student project** and was not built. What
*is* built mechanically implements "retraining strategy," "compare new
models before replacing," and "version every model, keep experiment
history" exactly as specified — a human (or a scheduled task, see the
deployment guide) runs `python retrain.py`, and the promotion gate,
versioning, and history are all real and automatic *within that run*.

### 11.3 Data-quality evaluation for future real-usage retraining

When MindRise accumulates enough real per-user prediction history to
retrain on real (not calibration-derived) outcomes, `retrain.py`'s data
argument can point at a Laravel-exported CSV of real
`exam_readiness_predictions` history instead of the current hybrid file —
no code change required. The data-quality bar for including a real row at
that point should mirror the OULAD/UCI mapping's own conservatism: only
students with ≥5 answered questions in a category before trusting that
category's accuracy as signal (the same `MIN_SAMPLE_SIZE` threshold
`WeakAreaWeightingService` already applies), and only predictions with a
genuine downstream outcome to validate against (a later real exam result,
once `exam_profiles` supports recording one).

## 12. Research Validation

### 12.1 Threats to validity

| Threat | Discussion |
|---|---|
| **Construct validity of the label** | For synthetic rows, "readiness" is defined by a documented composite heuristic (§3.4), not observed exam pass/fail — the model learns to predict *that heuristic*, which is a domain-expert approximation of readiness, not readiness itself. For real rows, the label is genuine (`final_result`/grade band), but is a *course outcome*, not a *government competitive-exam* outcome — MindRise's actual target domain. This is the single largest validity gap and is not resolved by this upgrade; it is narrowed (from 0% real-outcome grounding to 45.7%) but not closed. |
| **External validity / population match** | OULAD is UK distance-learning adults; UCI is Portuguese secondary-school students. Neither population matches MindRise's actual target users (Sri Lankan government-exam candidates, ages 20-30). The calibrated coefficients (§3.4) reflect *these* populations' engagement→outcome relationship, which may not transfer perfectly. |
| **Platform-only feature approximation** | §3.3's pseudo-theta derivation is a documented approximation, not a measurement — real rows' cognitive-category scores are generated, not observed. |
| **Temporal validity of the multi-output targets** | §9.3's first-half/second-half split uses a fixed module-length midpoint as the split point; a different midpoint choice would produce a different target definition, though the qualitative conclusions (risk classification is easy, score regression is hard) are unlikely to be split-point-sensitive. |
| **Synthetic-row Gaussian noise** | The composite-score heuristic (§3.4) injects noise so labels aren't perfectly deterministic from features — but the *noise distribution itself* (documented as Gaussian, SD=0.35 in z-score units) is a modelling choice, not derived from real label noise (which is unobservable for a heuristic label). |

### 12.2 Bias analysis

Using OULAD's real demographic fields (kept for analysis only, never as a
model feature — see §12.3), the calibrated model's real-data training
subset was checked for outcome disparity across:

- **Gender** (`gender`: M/F)
- **Disability status** (`disability`: Y/N)
- **Age band** (`age_band`: <=35 / 35-55 / 55<=)
- **Deprivation** (`imd_band` midpoint, a UK Index of Multiple Deprivation
  proxy)

**Real outcome disparity found** (share of students reaching the "ready"
label, i.e. Distinction; `data_pipeline/bias_fairness_report.py`, run on
all 32,593 real OULAD students):

| Group | n | "Ready" share |
|---|---|---|
| Gender: F | 14,718 | 9.5% |
| Gender: M | 17,875 | 9.1% |
| Disability: No | 29,429 | 9.5% |
| Disability: Yes | 3,164 | **7.0%** |
| Age: 0-35 | 22,944 | 8.1% |
| Age: 35-55 | 9,433 | 11.9% |
| Age: 55+ | 216 | 19.0% |
| Deprivation: most-deprived third | 14,020 | **6.7%** |
| Deprivation: mid third | 9,285 | 9.6% |
| Deprivation: least-deprived third | 8,177 | **12.2%** |

Gender shows near-parity (a 0.4 percentage-point gap). Disability status and
socioeconomic deprivation both show a real, non-trivial outcome gap (2.5pp
and 5.5pp respectively) — students with a disability, and students from the
most-deprived third of areas, reach "ready" at a meaningfully lower rate in
this real data. Age shows the largest apparent gap, but the 55+ group is
small (n=216) and its estimate is correspondingly less reliable. These
disparities are a property of the **real OULAD population and UK higher
education context**, not of MindRise or this model — reporting them
transparently (§12.3 explains why they cannot leak into the model as a
feature) is itself the correct research practice, not a flaw to fix by
adjusting the model.

### 12.3 Fairness-by-design decisions

Demographic fields (gender, disability, age band, deprivation) are
**excluded from the model's feature vector entirely** — `feature_mapping.py`
keeps them only as `_demographic_*`-prefixed columns, explicitly dropped
before any model ever sees the dataframe (see that file's `map_oulad()`/
`map_uci()`). This is a deliberate fairness-by-design choice: an "exam
readiness" prediction should never be able to directly condition on a
protected characteristic, even one correlated with real-world outcome
disparities the source data reflects (using such a field as a *feature*
would risk the model learning to reproduce the disparity as a decision
rule, not just report on it).

### 12.4 Privacy considerations

- **OULAD and UCI are both already fully anonymized** by their publishers
  (student IDs are dataset-internal integers with no link to real
  identities) — no additional anonymization step was needed or performed
  on the public data.
- **No real MindRise user data was used to train any model version shipped
  with this codebase** — the hybrid dataset is exclusively public data +
  documented synthetic generation; real MindRise predictions only ever
  flow *into* the platform's own `exam_readiness_predictions` history
  table for that individual student's own dashboard, never into a shared
  training file, under the current architecture.
- **Demographic fields are never sent to the FastAPI service or stored in
  a MindRise prediction record** — they exist only inside the offline
  training-data pipeline for the bias analysis in §12.2, and are dropped
  before training (§12.3).

### 12.5 Limitations

1. The core readiness label is only 45.7% grounded in real outcomes; the
   remainder is a documented but ultimately synthetic heuristic (§12.1).
2. Neither real dataset's population matches MindRise's actual target
   demographic.
3. Platform-only cognitive features for real rows are a structural-model
   approximation from pseudo-theta, not a measurement.
4. The two regression multi-output targets (§9.4) have modest R²
   (0.29–0.32) — useful as a directional signal, not as a precise
   point-forecast a student should treat as exact.
5. `days_until_exam` for real rows is a fixed constant (§3.3), not a
   genuine varying feature — real data can't inform how the model responds
   to exam-date urgency, only synthetic rows can.
6. The continual-learning promotion gate (§11) has not yet been exercised
   against a second real-data-driven training run — the first live model
   from this upgrade is, by construction, the first champion with nothing
   to have beaten yet.

### 12.6 Future improvements

1. Once MindRise accumulates real per-student `exam_readiness_predictions`
   history *paired with a real later exam outcome* (requires
   `exam_profiles` to support recording that outcome — not yet built),
   retrain on real MindRise labels directly, closing the largest validity
   gap in §12.1.
2. Investigate a Sri Lankan or South Asian government-exam-preparation
   learning-analytics dataset, if one becomes publicly available, to
   narrow the population-mismatch threat in §12.1.
3. Extend the temporal multi-output split (§9.3) to MindRise's own session
   history once volume permits, rather than relying on OULAD's module
   structure as a proxy for "first half / second half of a preparation
   period."
4. Revisit the consistency_score sign-flattening simplification (§3.4) with
   a calibration approach that preserves coefficient sign, once enough real
   MindRise data exists to validate which direction actually holds for
   this platform's own population.
